Q16. Relational Algebra, Tuple Relational Calculus, Domain Relational Calculus
Relations:
Opening (AccountNumber, OpenDate, OpeningBalance, TotalDeposit, TotalWithdrawal, ClosingBalance, ClosingBalanceDate, LastDepositDate, LastWithdrawalDate)
Deposit (AccountNumber, Date, Amount, Mode)
Withdrawal (AccountNumber, Date, Amount, Mode)
AccountHolder (AccountNumber, Name, BuildingNumber, AreaName, StreetNumber, CityCode, PinCode, StateCode)
Cities (CityCode, CityName, StateCode)
State (StateCode, StateName)
(a) Account Numbers with no deposits in Jan 2000:
Relational Algebra:
π AccountNumber (Opening) - π AccountNumber (σ Date ≥ '2000-01-01' AND Date ≤ '2000-01-31' (Deposit))
Tuple Relational Calculus:
{t | t ∈ Opening ∧ ¬∃d ∈ Deposit (d.AccountNumber = t.AccountNumber ∧ d.Date ≥ '2000-01-01' ∧ d.Date ≤ '2000-01-31')}
Domain Relational Calculus:
{<a\> | <a, o, ob, td, tw, cb, cbd, ldd, lwd> ∈ Opening ∧ ¬∃<a, dt, amt, m> ∈ Deposit (dt ≥ '2000-01-01' ∧ dt ≤ '2000-01-31')}
(b) Account Numbers with no deposits or withdrawals in Jan 2000:
Relational Algebra:
(π AccountNumber (Opening) - π AccountNumber (σ Date ≥ '2000-01-01' AND Date ≤ '2000-01-31' (Deposit))) ∩ (π AccountNumber (Opening) - π AccountNumber (σ Date ≥ '2000-01-01' AND Date ≤ '2000-01-31' (Withdrawal)))
Tuple Relational Calculus:
{t | t ∈ Opening ∧ ¬∃d ∈ Deposit (d.AccountNumber = t.AccountNumber ∧ d.Date ≥ '2000-01-01' ∧ d.Date ≤ '2000-01-31') ∧ ¬∃w ∈ Withdrawal (w.AccountNumber = t.AccountNumber ∧ w.Date ≥ '2000-01-01' ∧ w.Date ≤ '2000-01-31')}
Domain Relational Calculus:
{<a\> | <a, o, ob, td, tw, cb, cbd, ldd, lwd> ∈ Opening ∧ ¬∃<a, dt, amt, m> ∈ Deposit (dt ≥ '2000-01-01' ∧ dt ≤ '2000-01-31') ∧ ¬∃<a, dt, amt, m> ∈ Withdrawal (dt ≥ '2000-01-01' ∧ dt ≤ '2000-01-31')}
(c) Account Numbers and Names from ROHTAK with opening balance ≥ 9999:
Relational Algebra:
π AccountNumber, Name (σ CityName = 'ROHTAK' ∧ OpeningBalance ≥ 9999 (AccountHolder ⋈ Cities ⋈ Opening))
Tuple Relational Calculus:
{<a.AccountNumber, a.Name\> | a ∈ AccountHolder ∧ ∃c ∈ Cities (c.CityCode = a.CityCode ∧ c.CityName = 'ROHTAK') ∧ ∃o ∈ Opening (o.AccountNumber = a.AccountNumber ∧ o.OpeningBalance ≥ 9999)}
Domain Relational Calculus:
{<a, n\> | <a, n, bn, an, sn, cc, pc, sc> ∈ AccountHolder ∧ ∃<cc, 'ROHTAK', sc> ∈ Cities ∧ ∃<a, o, ob, td, tw, cb, cbd, ldd, lwd> ∈ Opening (ob ≥ 9999)}
(d) Cities of the state HARYANA:
Relational Algebra:
π CityName (σ StateName = 'HARYANA' (Cities ⋈ State))
Tuple Relational Calculus:
{c.CityName | c ∈ Cities ∧ ∃s ∈ State (s.StateCode = c.StateCode ∧ s.StateName = 'HARYANA')}
Domain Relational Calculus:
{<cn\> | ∃<cc, cn, sc> ∈ Cities ∧ ∃<sc, 'HARYANA'> ∈ State}
(e) Account Numbers from state HARYANA:
Relational Algebra:
π AccountNumber (AccountHolder ⋈ Cities ⋈ State)
Tuple Relational Calculus:
{a.AccountNumber | a ∈ AccountHolder ∧ ∃c ∈ Cities (c.CityCode = a.CityCode) ∧ ∃s ∈ State (s.StateCode = c.StateCode ∧ s.StateName = 'HARYANA')}
Domain Relational Calculus:
{<a\> | ∃<a, n, bn, an, sn, cc, pc, sc> ∈ AccountHolder ∧ ∃<cc, cn, sc> ∈ Cities ∧ ∃<sc, 'HARYANA'> ∈ State}
(f) City Names with no Account Holders:
Relational Algebra:
π CityName (Cities) - π CityName (AccountHolder ⋈ Cities)
Tuple Relational Calculus:
{c.CityName | c ∈ Cities ∧ ¬∃a ∈ AccountHolder (a.CityCode = c.CityCode)}
Domain Relational Calculus:
{<cn\> | ∃<cc, cn, sc> ∈ Cities ∧ ¬∃<a, n, bn, an, sn, cc, pc, sc> ∈ AccountHolder}
(g) Account Numbers with no transactions after opening:
Relational Algebra:
π AccountNumber (Opening) - π AccountNumber (Deposit) - π AccountNumber (Withdrawal)
Tuple Relational Calculus:
{o.AccountNumber | o ∈ Opening ∧ ¬∃d ∈ Deposit (d.AccountNumber = o.AccountNumber ∧ d.Date > o.OpenDate) ∧ ¬∃w ∈ Withdrawal (w.AccountNumber = o.AccountNumber ∧ w.Date > o.OpenDate)}
Domain Relational Calculus:
{<a\> | <a, o, ob, td, tw, cb, cbd, ldd, lwd> ∈ Opening ∧ ¬∃<a, dt, amt, m> ∈ Deposit (dt > o) ∧ ¬∃<a, dt, amt, m> ∈ Withdrawal (dt > o)}

Q17. Professor-Department-Committee Database
Relations:

(a) Professors in committees with Professor David:
π profname (committee ⋈ σ profname='David' (committee))
(b) Professors in all committees with Professor David:
π profname (committee) ÷ π commname (σ profname='David' (committee))
(c) Professors in exactly the same committees as Professor David:
(π profname (committee) ÷ π commname (σ profname='David' (committee))) ∩ π profname (σ commname IN (π commname (σ profname='David' (committee))) (committee))

π profname (σ deptname = (σ deptname (σ profname='David' (professor))) (professor))
(e) Departments with all professors in some committee:
π deptname (professor) ÷ π profname (committee)

Q21. Functional Dependencies and Normalization
(a) Given R(A, B, C, D) with FDs: A → B, B → C, C → D, find all candidate keys.
A → B, B → C, C → D means A → BCD.
Therefore, A+ = ABCD.
A is the only candidate key.
(b) Normalize R to 3NF.
A → B, B → C, C → D are transitive dependencies.
R1(A, B)
R2(B, C)
R3(C, D)
(c) Normalize R to BCNF.
R1(A, B)
R2(B, C)
R3(C, D) (BCNF is the same as 3NF here)

Q26. Database Architectures
Client-Server Architecture:
This architecture is best suited for airline reservations and ticket sales. The DBMS acts as a server, and numerous client applications (e.g., travel agency terminals, online booking websites) send requests to it.
Why it's better:
Centralized data management: Ensures consistency.
Scalability: Handles many concurrent users.
Security: Centralized control.
Why other architectures are not good:
Centralized: While centralized is good for data control, it is not optimized for high concurrent user access.
Distributed: Overkill for a single airline system. Increased complexity.
Parallel: Unnecessary overhead for this application.
Q27. Database Architectures (Explanation)
The Client-Server architecture is the most appropriate for airline reservations and ticket sales due to its ability to handle a large number of concurrent users, maintain data consistency, and provide centralized security.
In this scenario, numerous clients (travel agents, online users) need to access and modify the same database simultaneously. The Client-Server model allows for efficient handling of these requests.

Q30. Relational Algebra Query
(d) List all vertices reachable from a given vertex:
This query cannot be expressed by a relational algebra expression of constant length. Reachability requires a recursive or iterative process, which relational algebra, in its basic form, cannot handle. It would require a variable number of join operations, making it non-constant.
The other options can be solved with constant length relational algebra.
